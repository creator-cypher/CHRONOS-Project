# Chronos — Privacy & Ethics Impact Assessment

## 1. Executive Summary

Chronos is a context-aware ambient display system designed for domestic environments. This assessment evaluates the privacy implications of its multi-user authentication system and the ethical considerations of its algorithmic content curation — with particular focus on the **Kids Safety Filter** as a case study in Responsible AI.

---

## 2. Local Data Sovereignty

### 2.1 Architecture Decision: SQLite as Privacy Primitive

Chronos stores all user data — credentials, images, preferences, interaction logs, and AI-generated metadata — in a local SQLite database (`chronos.db`). This design is a deliberate privacy-by-architecture decision:

| Property | SQLite (Chronos) | Cloud Database |
|---|---|---|
| Data location | User's device | Remote server |
| Network required | No | Yes |
| Third-party access | Impossible | Possible |
| Data portability | Copy one file | Export/migration |
| Jurisdictional risk | None | Subject to hosting country laws |
| Breach surface | Physical device only | Network + server + supply chain |

**Key implication:** No user data — including login credentials, viewing patterns, or AI analysis results — ever leaves the user's local machine during normal operation. The system runs fully offline once images are uploaded.

### 2.2 Password Security

User passwords are hashed with **bcrypt** (work factor 12, salted) before storage. Bcrypt's adaptive cost function ensures that brute-force attacks remain computationally infeasible even as hardware improves. The plaintext password is never written to disk, logged, or transmitted.

### 2.3 Image Storage

Images are uploaded to Cloudinary (a cloud CDN) for efficient delivery to the Streamlit frontend. This is the **only external data flow** in the system. For fully offline deployments, images can alternatively be stored on the local filesystem. The Cloudinary integration is optional — removing it requires only changing the upload handler.

### 2.4 AI Analysis Data Flow

When Gemini Vision API analyses an image, the image URL is sent to Google's API endpoint. The returned metadata (mood, tags, description, colours) is stored locally in SQLite. No user identity information is included in the API request. This represents a **one-way, anonymised data flow** — Google receives an image URL but has no knowledge of which user uploaded it or how the system uses the analysis.

## 3. The Kids Safety Filter — Responsible AI

### 3.1 Design Philosophy

The Kids Safety Filter implements **algorithmic content curation** — the system actively shapes what content a child sees based on pre-defined safety heuristics. This raises important questions about:

- **Who decides** what is "safe" for children?
- **How transparent** is the filtering logic?
- **What biases** might the filter introduce?

Chronos addresses these through **deterministic, auditable rules** rather than opaque ML models.

### 3.2 Implementation

The filter operates at two levels:

**Level 1 — Mood Blocking** (coarse filter):
```
BLOCKED_MOODS = {melancholic, mysterious}
PREFERRED_MOODS = {joyful, energetic, calm}
```

Images whose AI-detected `primary_mood` falls into `BLOCKED_MOODS` are excluded from the candidate pool entirely. This prevents dark, unsettling, or emotionally complex imagery from being selected.

**Level 2 — Tag-Based Filtering** (fine filter):
```
BLOCKED_TAGS = {dark, melancholic, complex, moody, gloomy, noir,
                dramatic, horror, violent, scary, eerie, sinister,
                disturbing, unsettling, provocative}

SAFE_TAGS    = {bright, cartoon, nature, educational, colorful,
                playful, animal, fun, cheerful, sunshine, friendly,
                whimsical, landscape, flower, garden, ocean, sky, rainbow}
```

Each candidate image's semantic tags (generated by Gemini Vision) are checked against both sets. Images with any blocked tag are excluded. Images with safe tags receive a scoring boost (+0.03 per matching tag), making bright, educational, and nature-themed content more likely to be displayed.

### 3.3 Ethical Analysis

#### Strengths

1. **Transparency:** The filter rules are hardcoded constants, not learned parameters. Any developer or auditor can read the exact list of blocked/safe tags and understand the system's behaviour completely.

2. **Determinism:** Given the same image tags, the filter always produces the same result. There is no probabilistic component that could introduce unpredictable behaviour.

3. **Fail-safe:** If the filter removes all candidates, it falls back to images with preferred moods (joyful, energetic, calm) rather than showing nothing. The system degrades gracefully.

4. **No behavioural tracking of children:** The system does not build a preference profile of a child's viewing patterns for targeting purposes. The filter applies uniformly regardless of interaction history.

#### Limitations

1. **Cultural bias in tag definitions:** The distinction between "safe" and "unsafe" content is culturally situated. The current tag lists reflect Western cultural norms — what is considered "scary" or "complex" varies across cultures.

2. **Dependence on AI accuracy:** The filter trusts Gemini's mood and tag classifications. If the AI misclassifies a disturbing image as "cheerful," the filter will not catch it. This is a fundamental limitation of any system that relies on upstream AI classification.

3. **Binary categorisation:** The filter treats content as either safe or unsafe, with no gradation. A more nuanced system might consider the child's age, parental preferences, or context (time of day, educational setting).

4. **Over-filtering risk:** The broad blocked-tag list may exclude genuinely appropriate content. For example, a photograph of a sunset over a moody landscape might be tagged "moody" and filtered despite being perfectly child-safe.

### 3.4 Recommendations for Future Work

1. **Parental controls:** Allow parents to customise the blocked/safe tag lists per child profile, rather than relying on hardcoded defaults.

2. **Age-graded filtering:** Replace the binary Kids/Standard distinction with age ranges (3-6, 7-12, 13+) with progressively relaxed filtering.

3. **Audit logging:** Log which images were filtered and why, so parents can review the system's decisions.

4. **Human-in-the-loop:** Provide a mechanism for parents to manually approve/block specific images, overriding the algorithmic filter.

## 4. Multi-User Privacy Boundaries

### 4.1 Data Isolation

Each authenticated user's data is scoped by their `user_id`:

- **Images:** Each user sees only images they uploaded
- **Context logs:** Selection history is per-user
- **Preferences:** Mood, sensitivity, and override settings are per-user
- **Interactions:** Likes and skips are scoped to the user's own images

This prevents one household member from viewing another's interaction history, uploaded images, or display preferences.

### 4.2 Profile Type as Functional Role

The `profile_type` field (Standard, Kids, Professional) is not a privacy classification — it is a **functional role** that determines:

1. Which content filtering rules apply (Kids → safety filter)
2. Which visual theme is rendered (Kids → Sky Blue, Professional → Silver)
3. Which scoring adjustments are made (Kids → mood override to joyful)

The profile type does not affect data access controls — all users have the same read/write permissions over their own data.

### 4.3 Session Security

Authentication state is stored in Streamlit's `session_state`, which is:
- Server-side (not stored in cookies or localStorage)
- Cleared on browser tab close
- Not shared across browser tabs
- Not persistent across server restarts

This means a user must re-authenticate after each session, which is appropriate for a shared household device.

## 5. GDPR & Data Protection Considerations

### 5.1 Data Minimisation

Chronos collects only the data necessary for its function:
- Username, display name, email (optional), password hash
- Uploaded images and their AI-generated metadata
- Interaction feedback (like/skip) for scoring improvement
- Context logs for the reasoning overlay

No telemetry, analytics, or usage tracking data is collected or transmitted.

### 5.2 Right to Erasure

Since all data resides in a single SQLite file, complete data erasure is achieved by deleting `chronos.db`. Per-user erasure can be implemented by deleting rows matching a `user_id` across all tables.

### 5.3 Data Portability

The SQLite database is a single, self-contained file that can be copied, backed up, or migrated without any external dependencies. This is the simplest possible implementation of data portability.

---

## 6. Conclusion

Chronos demonstrates that **privacy and personalisation are not mutually exclusive**. By keeping all data local, using deterministic content filters, and implementing transparent algorithmic rules, the system achieves context-aware ambient display functionality without compromising user privacy.

The Kids Safety Filter, while imperfect, represents a practical implementation of Responsible AI principles — prioritising transparency, determinism, and fail-safe behaviour over opaque ML-based content moderation. Its limitations (cultural bias, AI dependence, binary categorisation) are well-understood and can be addressed through the recommended improvements.

The local-first architecture ensures that Chronos can serve as a case study for **"Privacy in the Home"** — a key theme in ambient computing research — demonstrating that intelligent, adaptive systems can operate entirely within the user's domain of control.
