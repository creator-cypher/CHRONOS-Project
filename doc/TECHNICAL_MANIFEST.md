## 1. System Architecture Overview

### 1.1 Modular Design Principles

The CHRONOS system is built on a three-layer architecture derived from CONTESS (Context-Aware Edge Computing Framework) principles:

```
┌─────────────────────────────────────────────────────────┐
│               USER INTERFACE LAYER (Streamlit)          │
│           • Real-time image display (cinematic)         │
│           • Interactive control dashboard               │
│           • Reasoning overlay (transparency)            │
└────────────────────┬────────────────────────────────────┘
                     │
┌────────────────────▼────────────────────────────────────┐
│        DECISION ENGINE LAYER (logic/engine.py)          │
│      • Weighted scoring (Context × Metadata)            │
│      • Mood & time compatibility matrices               │
│      • Seasonality factor integration                   │
│      • Confidence score calculation                     │
└────────────────────┬────────────────────────────────────┘
                     │
        ┌────────────┴────────────┐
        │                         │
┌───────▼─────────────┐  ┌────────▼──────────────┐
│  CONTEXT LAYER      │  │  METADATA LAYER       │
│  (logic/context.py) │  │  (services/vision.py) │
│ • Time periods      │  │ • AI image analysis   │
│ • Mood detection    │  │ • Tag extraction      │
│ • Seasonal factors  │  │ • Colour analysis     │
│ • Data validation   │  │ • Quality scoring     │
└────────────────────┘  └───────┬──────────────┘
                                 │
                    ┌────────────▼─────────────┐
                    │   PERSISTENCE LAYER     │
                    │   (database/)           │
                    │ • SQLite (6 tables)     │
                    │ • WAL mode (concurrency)│
                    │ • Audit trail logging   │
                    └─────────────────────────┘
```

### 1.2 Design Rationale

**Why Modular?**
- **Separation of Concerns:** Context, Decision, and Metadata are independently testable and replaceable
- **Scalability:** Each layer can be optimized or upgraded without affecting others
- **Transparency:** Audit logs enable full traceability of every decision
- **Adaptability:** Weights, compatibility matrices, and thresholds can be tuned without code changes

## 2. Data Layer (Database & Persistence)

### 2.1 Schema Design

**SQLite Database:** `chronos.db`
**Concurrency Mode:** WAL (Write-Ahead Logging) with foreign key constraints enabled

#### Table 1: `images`
Primary data store for display images with AI-enriched metadata.

| Column | Type | Notes |
|--------|------|-------|
| `id` | TEXT (UUID) | Primary key |
| `title` | TEXT | User-friendly name |
| `image_path` | TEXT | Local file path (base64 encoded in CSS) |
| `image_url` | TEXT | External HTTPS URL (fallback) |
| `ai_description` | TEXT | Poetic description from Gemini |
| `primary_mood` | TEXT | One of: calm, energetic, melancholic, joyful, mysterious, neutral |
| `optimal_time` | TEXT | One of: dawn, morning, afternoon, evening, night, any |
| `base_score` | REAL | [0.0, 1.0] quality metric from Gemini |
| `dominant_colors` | JSON | Array of hex colour codes |
| `is_analyzed` | INTEGER | Boolean flag (0/1) |
| `is_active` | INTEGER | Soft-delete flag (1=active, 0=archived) |
| `display_count` | INTEGER | Total displays (recency tracking) |
| `last_displayed` | TEXT | ISO timestamp (NULLS FIRST in queries) |
| `uploaded_at` | TEXT | Creation timestamp |

**Indexes:**
- `idx_mood_time`: (primary_mood, optimal_time) — Decision Engine scoring
- `idx_active`: (is_active, is_analyzed) — Candidate pool filtering
- `idx_log_ts`: context_logs(timestamp DESC) — History queries

#### Table 2: `image_tags`
Semantic labels generated by Gemini Vision API.

| Column | Type | Notes |
|--------|------|-------|
| `id` | INTEGER | Auto-increment |
| `image_id` | TEXT (FK) | Reference to images.id |
| `name` | TEXT | Tag label (e.g., "sunset", "calm", "indoor") |
| `category` | TEXT | One of: subject, mood, time, color, style, technical |
| `confidence` | REAL | [0.0, 1.0] from Gemini |
| **UNIQUE** | (image_id, name) | Prevents duplicate tags per image |

#### Table 3: `user_preferences`
Singleton record for user settings and system configuration.

| Column | Type | Notes |
|--------|------|-------|
| `id` | INTEGER | Always = 1 (singleton) |
| `preferred_mood` | TEXT | User's global mood preference |
| `sensitivity` | TEXT | One of: low, medium, high (weight profile selector) |
| `time_mood_map` | JSON | Per-period mood overrides (e.g., {"evening": "joyful"}) |
| `override_active` | INTEGER | Boolean: is manual selection active? |
| `override_image_id` | TEXT | Which image is manually selected |
| `recency_weight` | REAL | Penalty multiplier (default: 0.2) |
| `updated_at` | TEXT | Last modification timestamp |

#### Table 4: `context_logs`
Immutable audit trail of every selection decision.

| Column | Type | Notes |
|--------|------|-------|
| `id` | INTEGER | Auto-increment |
| `timestamp` | TEXT | ISO timestamp of decision |
| `time_period` | TEXT | Active time period (dawn/morning/etc.) |
| `detected_mood` | TEXT | Auto-detected mood from time |
| `selected_image_id` | TEXT (FK) | Which image was chosen |
| `selection_score` | REAL | Final weighted score [0.0, 1.0] |
| `score_breakdown` | JSON | {time, mood, preference, quality, recency, interaction} |
| `matched_tags` | JSON | Array of tags that contributed to match |
| `reasoning_text` | TEXT | Human-readable explanation |
| `was_override` | INTEGER | Boolean: manual override? |

#### Table 5: `image_interactions`
User feedback signals for dynamic preference learning.

| Column | Type | Notes |
|--------|------|-------|
| `id` | INTEGER | Auto-increment |
| `image_id` | TEXT (FK) | Which image |
| `interaction` | TEXT | One of: "like", "skip" (CHECK constraint) |
| `timestamp` | TEXT | When feedback was given |

#### Table 6: `display_config`
Singleton configuration for display behavior and UI parameters.

| Column | Type | Notes |
|--------|------|-------|
| `id` | INTEGER | Always = 1 (singleton) |
| `poll_interval_seconds` | INTEGER | Auto-refresh frequency (default: 300s) |
| `transition_duration_ms` | INTEGER | Image fade duration (default: 1500ms) |
| `show_reasoning_overlay` | INTEGER | Boolean |
| `overlay_auto_hide_seconds` | INTEGER | Reasoning card auto-hide delay |
| `night_brightness` | REAL | Brightness filter at night (default: 0.7) |
| `updated_at` | TEXT | Last modification |

## 3. Context Layer (logic/context.py)

### 3.1 Time Period Classification

Maps local system time → symbolic time period + mood association.

```python
TIME_PERIODS = [
    (5,  8,  "dawn",      "calm"),
    (8,  12, "morning",   "energetic"),
    (12, 17, "afternoon", "joyful"),
    (17, 21, "evening",   "calm"),
    (21, 24, "night",     "mysterious"),
    (0,  5,  "night",     "mysterious"),  # Overlap handled gracefully
]
```

**Edge Cases Handled:**
- Midnight (hour=0): Checked in second "night" range
- Fallback: If no match, defaults to "night"/"mysterious"
- Validation: All moods sanitized to canonical set in `get_current_context()`

### 3.2 Seasonal Context

Colour temperature and mood associations by season:

| Season | Dominant Mood Boost | Rationale |
|--------|-------------------|-----------|
| Winter | Calm (+10%), Mysterious (+15%) | Shorter days, cooler tones |
| Spring | Joyful (+15%), Energetic (+10%) | Renewal, warming |
| Summer | Energetic (+20%), Joyful (+15%) | Long days, bright light |
| Autumn | Mysterious (+15%), Melancholic (+10%) | Transitional, warm-cool |

### 3.3 Context Validation (Fixed in Audit)

**Issue:** Overridden mood values could be invalid, causing silent scoring degradation.
**Fix:** Added runtime validation to ensure all moods are from canonical set {calm, energetic, melancholic, joyful, mysterious, neutral}.

## 4. Decision Engine (logic/engine.py)

### 4.1 Scoring Algorithm

**Formula:**
```
Score = w_time × T(period, img.optimal_time)
      + w_mood × M(user_mood, img.mood, detected_mood, season)
      + w_pref × P(img.mood, img.tags, target_mood)
      + w_qual × Q(img.base_score)
      - w_recency × R(recency_penalty)
      + I(user_interactions)
```

### 4.2 Weight Profiles (Sensitivity Levels)

User can configure system responsiveness:

| Sensitivity | Time | Mood | Preference | Quality | Recency |
|-------------|------|------|-----------|---------|---------|
| **Low** | 0.20 | 0.15 | **0.40** | 0.15 | 0.10 |
| **Medium** | 0.35 | 0.25 | 0.20 | 0.10 | 0.10 |
| **High** | 0.40 | 0.30 | **0.10** | 0.10 | 0.10 |

**Interpretation:**
- **Low:** Trust user mood preference most (e.g., "I always want calm images")
- **Medium:** Balanced; context + mood influence equally
- **High:** Follow time-of-day closely (e.g., energetic in morning, calm at night)

### 4.3 Mood Compatibility Matrix

Symmetric matrix defining how well each image mood matches target moods:

```python
MOOD_COMPAT = {
    "calm": {"calm": 1.0, "melancholic": 0.5, "neutral": 0.4},
    "energetic": {"energetic": 1.0, "joyful": 0.6, "neutral": 0.3},
    # ... (complete matrix in engine.py)
}
```

**Design:** Scores represent "mood distance" — perfect match = 1.0, completely opposite ≈ 0.

### 4.4 Confidence Score Calculation

**New Feature (Audit Fix):** Display confidence % to user.

```
Confidence(%) = Base(0-60%) + TagBonus(0-30%)
              = (score × 60) + min(30, matched_tags × 5)
```

**Interpretation:**
- 90%+ = Multiple tags match, high total score → Trustworthy selection
- 60-70% = Some tags, moderate score → Reasonable choice
- <60% = Few tags, low score → Fallback selection (should be rare)

### 4.5 Seasonality Integration (New Feature)

Boosts mood compatibility by 5-20% depending on season:

```
m_score = (0.60 × pref_compat + 0.40 × detected_compat) × seasonal_boost
```

**Example:** In summer, "energetic" images get +20% bonus to mood score regardless of other factors.

### 4.6 Interaction Signal Processing

User feedback shapes future selections:

```
interaction_delta = max(-0.15, min(0.15, (likes - skips) × 0.05))
final_score = max(0.0, min(1.0, weighted_sum + interaction_delta))
```

**Limits:** ±0.15 cap prevents single feedback from dominating.

## 5. Metadata Layer (services/vision.py)

### 5.1 Gemini 2.5-Flash Integration

Calls Google Generative AI API with structured JSON prompt.

**Model:** `gemini-2.5-flash`
**Input:** Image (local file, HTTPS URL, or raw bytes)
**Output:** Structured JSON with:
- Poetic description (2-3 sentences)
- Primary mood (canonical value)
- Optimal display time (canonical value)
- Base quality score [0.0, 1.0]
- Dominant colour hex codes
- 5-10 semantic tags (name, category, confidence)

### 5.2 Tag Categories

| Category | Purpose | Examples |
|----------|---------|----------|
| **subject** | What is in the image | "sunset", "landscape", "person" |
| **mood** | Emotional tone (validated) | "calm", "joyful", "mysterious" |
| **time** | When it's appropriate | "morning", "night", "any" |
| **color** | Dominant hues | "warm", "blue-toned", "vibrant" |
| **style** | Artistic quality | "minimalist", "cinematic", "vintage" |
| **technical** | Image properties | "high-contrast", "bokeh", "macro" |

### 5.3 Error Handling

**Graceful Degradation:**
- No API key → Returns `AnalysisResult(success=False, ...)`
- JSON parse failure → Logs error, returns empty result
- Network timeout → Caught, returns error message (never crashes)
- Invalid image → Proper mime type detection

- Update visual feedback every 250ms

## 7. Evaluation & Metrics

### 7.1 Trackable Metrics (context_logs table)

1. **Selection Consistency:** How stable are decisions under repeated context?
2. **Tag Matching Rate:** % of matched tags vs. total available
3. **Interaction Responsiveness:** Does system adapt to like/skip feedback?
4. **Confidence Distribution:** Are confident selections correlated with positive user feedback?
5. **Recency Balance:** Does system avoid repeating recent images appropriately?

### 7.2 Comparison to Baseline

**Adaptive (CHRONOS):**
- AI-weighted selection based on time, mood, seasonality
- Confidence score visible
- User feedback integrated

**Baseline (app_baseline.py):**
- Random image rotation
- No AI analysis
- Manual prev/next/shuffle controls

**Evaluation Rubric:**
- Consistency under repeated contexts
- Perceived relevance vs. baseline
- User perceived usefulness
- System reliability (no crashes on edge cases)
